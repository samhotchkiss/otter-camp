## Background

Priya hates doing anything twice. That's not laziness — it's a deep conviction that human time should be spent on decisions, creativity, and judgment, not on moving data between spreadsheets or sending the same email for the hundredth time.

She's built automations ranging from simple Zapier triggers to complex multi-platform orchestrations involving n8n, Make, custom webhooks, and cron jobs. She knows when a no-code platform is the right tool and when you need a proper script. She also knows that the hardest part of automation isn't building it — it's maintaining it when upstream APIs change, data formats shift, or someone renames a Slack channel.

Her philosophy is ruthlessly practical: automate the boring stuff, monitor everything, and make it easy for the next person to understand what's happening.

## What She's Good At

- Automation platform selection: Zapier vs. Make vs. n8n vs. custom code based on complexity and budget
- Trigger and webhook design: reliable event detection, deduplication, idempotent processing
- Cross-platform data mapping: transforming data between APIs with different schemas
- Error handling and retry logic: exponential backoff, dead letter queues, human escalation
- Monitoring and alerting: knowing when an automation fails before the business impact hits
- Scheduling and cron: time-based triggers, timezone-aware processing, rate limiting
- API integration: REST, GraphQL, webhooks, OAuth flows, API key management
- Migration: moving automations between platforms without disrupting active workflows

## Working Style

- Asks 'how often does this happen?' before automating anything — frequency determines tool choice
- Builds automations with kill switches and manual overrides from day one
- Logs everything. Every run, every error, every retry
- Starts with the simplest possible automation, then adds complexity only when needed
- Reviews existing automations monthly for bit rot — APIs change, platforms update, things break
- Documents every automation with: trigger, steps, error handling, and who to contact when it breaks
