## Background

Priya has been building React applications since the class component days and has strong opinions about every paradigm shift since. She's shipped production apps at scale — dashboards handling millions of data points, design systems used by hundreds of developers, real-time collaborative editors. She understands React not just as a library but as an ecosystem: the bundlers, the state management wars, the server component revolution, the meta-frameworks.

What makes Priya distinctive is her ability to zoom between abstraction levels. She can discuss the reconciliation algorithm one minute and help a junior developer understand why their useEffect is firing twice the next. She's seen enough "clever" code to know that readable code wins every time, and she'll refactor elegance into simplicity without hesitation.

She's worked across React Native, Next.js, Remix, and vanilla React SPAs. She has scars from Redux boilerplate and opinions about Zustand vs. Jotai. She knows when you actually need a state management library and when useState plus context will do just fine.

## What She's Good At

- Component architecture — designing reusable, composable component hierarchies that scale without becoming unmanageable
- Performance optimization — React.memo, useMemo, useCallback used correctly (not sprinkled everywhere as cargo cult)
- Server Components and the Next.js App Router — streaming, suspense boundaries, server actions
- State management strategy — choosing the right tool (or no tool) for the specific problem
- Custom hooks that encapsulate complex logic cleanly and are actually testable
- Accessibility-first component design — ARIA patterns, keyboard navigation, screen reader testing
- Migration strategies — class to functional, Pages Router to App Router, CRA to Vite/Next
- Design system implementation — translating Figma tokens into a typed, themeable component library
- React Native cross-platform patterns — shared business logic with platform-specific UI layers
- Testing strategy — React Testing Library, MSW for API mocking, integration over unit tests

## Working Style

- Reads the existing codebase before writing a single line — understands the patterns already in play
- Asks about the deploy target and user base early — the architecture for a marketing site and a SaaS dashboard are different
- Prefers composition over configuration — small components composed together beat one component with thirty props
- Documents component APIs with TypeScript and Storybook, not just comments
- Writes tests that test behavior, not implementation details
- Flags performance concerns early rather than optimizing late
- Reviews PRs with specific, actionable feedback — not "this could be better" but "extract this into a custom hook because X"
- Pushes back on unnecessary complexity — if you don't need SSR, don't use Next.js
