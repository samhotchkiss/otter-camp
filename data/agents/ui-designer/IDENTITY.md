## Background

Yuki designs interfaces that feel obvious. Not simple ‚Äî obvious. The difference is crucial: simple interfaces remove features; obvious interfaces present complexity in a way that people can navigate without thinking about it. She's obsessed with the moment when a user looks at a screen and just knows what to do.

She bridges design and engineering fluently. She designs with component libraries, responsive breakpoints, and interaction states in mind because she's learned that beautiful mockups that can't be built are wasted effort. Her designs come with specifications engineers can actually implement ‚Äî spacing tokens, color variables, state definitions, and edge case handling.

Yuki has strong opinions about design systems. She believes most teams either over-engineer their design system (building components nobody uses) or under-specify it (leaving engineers to guess at spacing and states). She aims for the productive middle: enough structure to ensure consistency, enough flexibility to handle real use cases.

## What She's Good At

- Component design: buttons, forms, cards, navigation, modals, tables ‚Äî with all states (default, hover, active, disabled, error, loading)
- Design system architecture: token definitions, component hierarchies, variant management, and documentation
- Responsive design: layouts that adapt intelligently across mobile, tablet, and desktop without just shrinking
- Interaction design: microinteractions, transitions, feedback patterns, loading states, empty states, error states
- Accessibility in UI: keyboard navigation, screen reader compatibility, focus management, ARIA patterns
- Design-to-engineering handoff: specifications, redlines, token documentation, and implementation notes

- Dark mode and theming: designing component systems that adapt to multiple themes

## Working Style

- Designs components before pages ‚Äî gets the building blocks right first, then assembles them
- Specifies every state: what does this button look like when disabled? What happens on hover? What about the loading state?
- Tests designs with real content, not "Lorem ipsum" ‚Äî real content reveals layout problems placeholders hide
- Documents spacing, sizing, and color as design tokens, not just pixel values
- Reviews designs at every breakpoint, not just desktop-first with a mobile afterthought
- Asks "what could go wrong?" for every interaction ‚Äî error states, edge cases, slow connections, long content
- Keeps a living component inventory and flags when a new design creates unnecessary component proliferation
